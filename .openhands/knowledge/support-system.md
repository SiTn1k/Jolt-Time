# Jolt Time — Support System and Player Assistance

## Overview

The Jolt Time support ecosystem provides players with accessible, responsive, and respectful assistance whenever they need help. Support systems are designed to resolve issues quickly while maintaining a friendly, professional experience that makes players feel valued and heard.

**Core Philosophy:** Support exists to help players, not to create barriers. Every interaction should leave players feeling respected, informed, and confident their concerns matter.

## 2. Ticket System Philosophy

The ticket system empowers players to manage their support experience independently while ensuring no issue goes unaddressed.

### Create Support Requests

**Players can submit support requests for any concern:**

- Simple submission form with category selection
- Description field for detailed explanation
- Evidence attachment capability (screenshots, logs)
- Priority indication for urgent issues
- Automatic confirmation with ticket number
- Expected response time notification

### View Request History

**Players have complete visibility into their support history:**

- List of all submitted tickets
- Current status of each request
- Full conversation thread
- Response timestamps
- Resolution summary
- Easy re-opening of closed tickets if needed

### Receive Status Updates

**Players stay informed throughout the resolution process:**

- Automated status change notifications
- Staff responses delivered promptly
- Resolution confirmation when complete
- Option to add additional information anytime
- No status updates without new activity

### Review Solved Cases

**Post-resolution transparency:**

- Complete resolution summary provided
- Steps taken to resolve the issue
- Any compensation or actions applied
- Rating opportunity for support experience
- Option to reopen if issue persists

### Ticket Lifecycle

1. **Submission** — Player submits ticket with category and description
2. **Acknowledgment** — Automated response confirming receipt
3. **Triage** — Support team reviews and assigns priority
4. **Investigation** — Staff researches and gathers information
5. **Response** — Player receives updates and questions answered
6. **Resolution** — Issue addressed and confirmed
7. **Closure** — Ticket marked complete
8. **Feedback** — Optional satisfaction rating

## 3. Ticket Statuses

Clear status indicators keep players informed about their request progress.

### Open

**Initial state when ticket is submitted:**

- Ticket received and queued for review
- Automated acknowledgment sent
- Player can add additional information
- Internal triage in progress

### In Review

**Support team is actively working on the issue:**

- Assigned to support staff member
- Investigation underway
- Player may receive clarifying questions
- Progress updates provided when significant

### Waiting for Response

**Resolution requires player input or action:**

- Clarification needed from player
- Additional evidence requested
- Player action required to proceed
- Deadline reminder sent if needed

### Resolved

**Issue has been addressed:**

- Solution implemented or provided
- Player notified of resolution
- Summary of actions taken included
- Ticket remains viewable for reference
- Player can reopen if issue persists

### Closed

**Ticket is finalized:**

- No further action required
- Solution confirmed by player or timeout
- Archived for future reference
- Can be reopened by player if needed
- Maintains complete history

## 5. Evidence Philosophy

Evidence requirements balance thorough investigation with player convenience.

### Simple Evidence Requirements

**Players should not be burdened with excessive documentation:**

- Maximum three screenshots per report
- Basic description of what happened
- When possible, approximate timestamps
- No need for lengthy written statements
- Clear, concise, informative submissions

### Acceptable Evidence Types

**Screenshots**

- In-game screenshots preferred
- Clear and readable content
- Relevant context visible
- Date/time stamps helpful but not required
- Multiple angles if helpful

**Descriptions**

- What happened in plain language
- Who was involved (usernames or character names)
- Where it occurred (location, mode, context)
- When it happened (if known)
- Why it concerns the reporter

**Timestamps**

- Approximate time is sufficient
- Timezone not required
- Relative time accepted ("about an hour ago")
- Multiple reports can reference same incident

### Evidence Handling

**Support staff handle evidence responsibly:**

- Evidence viewed only for investigation purposes
- Screenshots stored securely and briefly
- Personal information in evidence protected
- Evidence deleted after case resolution
- No evidence shared with other players

## 10. Privacy Philosophy

Support staff handle player information with the highest respect for privacy.

### Only Necessary Access

**Staff access what they need, nothing more:**

- Minimum data principle for all investigations
- Specific purpose for each data access
- No browsing player data without reason
- Justified need required for elevated access
- Automatic access logging

### Respect Player Privacy

**Player privacy is paramount:**

- No sharing of player information
- Internal discussions need-to-know basis
- Evidence handled confidentially
- Personal details not logged unnecessarily
- Communications kept private

### Avoid Unnecessary Data Exposure

**Protect sensitive information:**

- Player identity protected in reports
- Financial details never exposed
- Location data not tracked
- Private messages never accessed
- Profile data only as needed

### Player Rights

**Players maintain control over their data:**

- Request data about data access
- Understand who viewed their information
- Data export capabilities
- Appropriate deletion options
- Transparent privacy policies
